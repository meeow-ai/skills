## 判断标准：什么值得沉淀为 skill

**值得沉淀：**
- 在多个不同项目或场景中都能复用的做法
- 需要反复向 Claude 解释的偏好或约定
- 踩过坑之后总结出的反直觉规则
- 节省大量来回沟通的提示模板或流程

**不值得沉淀：**
- 只针对某个项目的一次性配置
- 当前对话结束后不再需要的临时指令
- 能从代码或文档中直接推断出来的信息

## 文件格式规范

每个 skill 文件必须以 YAML frontmatter 开头，包含以下字段：

```
---
name: <与文件名一致的标识符>
trigger: <一句话说明：在什么情况下应该读这个文件>
last_updated: <YYYY-MM-DD>
---
```

正文使用 Markdown，结构自由，但需直接切入要点，不写废话。

## 更新时机

**新增 skill：**
当某个做法在两个以上不同场景中被验证有效，或预见到会被反复用到时，立即创建文件并在 `CLAUDE.md` 索引中添加一行。

**修改现有 skill：**
- 发现原有规则不够准确或出现例外时
- 有更好的替代做法时（直接覆盖，不保留旧版）
- 更新后同步修改 `last_updated` 字段

## Commit message 格式

```
skill(<名称>): <一句话描述变更内容>
```

示例：
- `skill(meta): 初始化 skills 维护规范`
- `skill(code-review): 新增 PR 审查检查清单`
- `skill(prompting): 更新链式思考触发条件`
